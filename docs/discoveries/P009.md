## Code Reality Map (cho Architect P010+)

### Deviations from phiếu spec

**SD-1 [mechanical] — `inbox::insert_rows` call-site: 3 params, phiếu showed 2**

Phiếu spec (pipeline outline step 7 + Task 1 code block) showed:
```rust
inbox::insert_rows(&inbox_content, &kept)
```

Real signature (`src/inbox.rs:66-70`):
```rust
pub fn insert_rows(
    content: &str,
    rows: &[AdvisoryRow],
    path: &Path,
) -> Result<(String, usize), InboxError>
```

Fix applied: pass `&inbox_path` as third argument. This was present since P006 — phiếu had a copy error. No API change. Recorded in Debate Log Turn 1 as `[mechanical]` — self-resolved.

**SD-2 [mechanical] — `state-3ids.json` fixture timestamp in the future relative to test machine clock**

`state-3ids.json.last_scan_at = "2026-05-28T09:51:35Z"` but test machine's `Utc::now()` was `"2026-05-28T05:07:39Z"` (machine running ~5h behind UTC fixture timestamp). Assertion `post.last_scan_at > pre.last_scan_at` would fail when reusing the fixture.

Fix applied: Test A uses inline `serde_json::json!` state with `"last_scan_at": "2026-01-01T00:00:00Z"` (safely in the past). IDs mirror `state-3ids.json` content for consistent overlap semantics. No fixture file modified.

Lesson: Integration tests that assert timestamp ordering must use inline fixtures with past timestamps, not pre-built fixtures whose `last_scan_at` may be in the future on test machines.

**SD-3 [mechanical] — Clippy: `doc_lazy_continuation` in test doc comment**

`///   Expected: appended=1, ...` on line 44 triggered `clippy::doc_lazy_continuation` ("doc list item without indentation"). Fixed by adding blank line + reformatting to proper list syntax.

**SD-4 [mechanical] — Clippy: `needless_borrows_for_generic_args` in Test B**

`.stdout(contains(&format!(...)))` triggered clippy. Fixed: `.stdout(contains(format!(...)))` (remove needless `&`).

## Observations (for future phiếu)

1. **P011 MCP tool** will need the same composite logic. Consider extracting a `mod scan` helper from `cli/scan_and_append.rs` at that time (in-scope for P011 — not done here per phiếu scope lock).

2. **Timestamp ordering in tests** — see SD-2. Any test asserting `post.last_scan_at > pre.last_scan_at` must use inline state with a deliberately old timestamp (e.g., `"2026-01-01T00:00:00Z"`), not a pre-built fixture file whose timestamp could be in the future on some machines.

3. **`InboxError::ParseRow` in scan-and-append dispatch** — this variant arm is required for exhaustive match compile-time enforcement, but `scan-and-append` never calls `parse_rows` so `ParseRow` will not surface at runtime. If future phiếu add `parse_rows` to scan-and-append, the exit code is already correctly mapped (→ 1).

4. **Test B uses `parse-report --input`** at runtime to extract row IDs from the fixture — this is robust to fixture changes but adds a subprocess invocation overhead in test. Acceptable for integration tests; if fixture is stable, could switch to inline known IDs (Tầng 2 future self-decide).

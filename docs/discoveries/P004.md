## Implementation Notes

### Commands::ParseReport dispatch — clippy fix (Tầng 2)

Phiếu V2 Task 3.2 showed a 3-branch if/else-if/else for the exit code mapping:

```rust
if e.is::<SentinelError>() { 1 }
else if e.is::<RowParseError>() { 2 }
else { 2 }
```

Clippy `-D warnings` catches `clippy::if_same_then_else`: the `RowParseError` branch and the fallthrough both return `2`, making the `else-if` redundant. Fixed by collapsing to 2 branches (Tầng 2 — local variable logic, self-decided):

```rust
if e.is::<crate::sentinel::SentinelError>() { 1 } else { 2 }
```

Behavior is identical: `SentinelError` → exit 1, all other errors → exit 2. Semantics unchanged from phiếu spec (ARCHITECTURE §1 exit codes preserved).

### `#![allow(dead_code)]` removal from `row.rs`

Removed as planned. After wire-in via `cli/parse_report.rs`, all public items in `row.rs` have concrete consumers in the binary code path (`parse_row` called from `run()`, `AdvisoryRow`/`Status`/`Severity` used in `parse_row` + serialized in JSON output). `cargo clippy --all-targets -- -D warnings` passes clean — no `dead_code` warning.

### `sentinel.rs` still has `#![allow(dead_code)}`

Noted in Turn 1 Challenge (Anchor #14 side-note). Phiếu explicitly scopes this to `row.rs` only (Constraint #11). After P004 wire-in, `sentinel::extract_block` IS consumed by `cli::parse_report::run()` — so the attribute on `sentinel.rs` is now stale. However, removing it was not in P004 scope. This should be cleaned up in a future phiếu (P005 or P006 as housekeeping). Tracking here for Architect.

### `serde_json::json!` key ordering

The phiếu spec showed output JSON with `"rows"` first, but `serde_json::json!` macro emits keys in alphabetical order for object literals. Actual output order is `advisories_found` → `rows` → `stack_scanned`. This does not affect correctness (JSON objects are unordered per spec) and the integration tests use `predicate::str::contains` substring matching — not exact string comparison. No issue for downstream consumers.

## rmcp 1.7.0 API Reality vs Architect Snapshots

### Confirmed matches

- `ServerHandler` trait at `handler/server.rs` — confirmed, 26+ provided methods.
- `stdio() -> (Stdin, Stdout)` — exact.
- `ServiceExt::serve` returning `Result<RunningService, ServerInitializeError>` — exact.
- `ServerInfo = type alias for InitializeResult` — exact.
- `Implementation` struct fields (`name`, `title?`, `version`, `description?`, `icons?`, `website_url?`, `#[non_exhaustive]`) — exact.
- `ServerCapabilities::builder().build()` = empty, `.enable_tools()` exists — exact.
- `ProtocolVersion` has `Default` impl — confirmed.

### Tầng 2 tuning applied

1. **`Implementation::new()` used instead of field-init:** `model.rs:1015` confirms `pub fn new(name: impl Into<String>, version: impl Into<String>)` exists. Used: `Implementation::new("advisory-inbox", env!("CARGO_PKG_VERSION"))`. Cleaner + forward-safe with `#[non_exhaustive]`.

2. **`ServerInfo::new(capabilities).with_server_info(impl)` builder pattern used:** Architect draft used field-init `ServerInfo { ... }`. Real impl uses builder: `ServerInfo::new(ServerCapabilities::builder().build()).with_server_info(Implementation::new(...))`. Both compile — builder is cleaner.

3. **`waiting()` return type confirmed as `Result<QuitReason, tokio::task::JoinError>`:** Architect phiếu noted uncertainty. `service.rs:545` confirms `Result<QuitReason, tokio::task::JoinError>`. `JoinError` implements `std::error::Error` → anyhow's `?` / `.context()` works directly. No adjustment needed.

4. **`ProtocolVersion::default()` returns LATEST = `"2025-11-25"`:** Server responds using LATEST in initialize response regardless of client's requested version. Test B uses `"2024-11-05"` to test backward-compat. Server echoes client's requested version in response (rmcp negotiation behavior).

## Code Pattern Decision Log

Phiếu draft used field-init for `Implementation` and `ServerInfo`. Worker used builder pattern after verifying rmcp API:

```rust
// Final impl (cleaner):
fn get_info(&self) -> ServerInfo {
    ServerInfo::new(ServerCapabilities::builder().build())
        .with_server_info(Implementation::new("advisory-inbox", env!("CARGO_PKG_VERSION")))
}
```

`Implementation::new()` confirmed at model.rs:1015. `ServerInfo::new(capabilities).with_server_info(impl)` confirmed at model.rs:865 + 881. Both patterns compile; builder is preferred for `#[non_exhaustive]` safety.

## Known Non-Issues

- `waiting()` returns `Result<QuitReason, JoinError>` not `()`. The `.context()` call is valid. Both `Cancelled` and `Closed` QuitReasons are treated as success (Ok variant); only JoinError (panic in spawned task) surfaces as error → exit 5 in main.
- `description` field on `Implementation` — exists in 1.7.0 (`Option<String>`). Not used via `new()` constructor (which sets it to None). Tầng 2: acceptable since server description is optional and not critical for P010 handshake.
